## 1. クライアント版数の整合チェック（★最優先の地雷除去）

**問題**: ロックステップは「両クライアントのエンジンが同一バイナリ」である前提。今後カード効果の修正を
デプロイするたび、「片方は新版・片方はブラウザキャッシュの旧版」の組で対戦すると**必ずdesync**する
（_customSeqの時と違い、ゲーム進行そのものが割れる）。カード実装は今も更新が続くため時限爆弾。

**設計**: ビルド時に版数（`import.meta.env` にgit hash か package.json version+ビルド時刻）を埋め、
`ready` に載せる → DO が両者の版数不一致なら `start` せず `error{code:'version_mismatch'}` を配信 →
ロビーに「アプリの版が違います。リロードしてください」表示。工数: 半日未満。

## 2. 持ち時間（ホスト設定・任意）※ユーザー要望の設計

**公式ルールの確認**: 公認大会のフロアルールは「制限時間を設ける場合、1対戦30分を推奨」で、
時間切れ時は**勝敗判定なしの両者敗北**（チェスクロック式ではなく、マッチ全体の時間）。
オンラインで「両者敗北」は体験が悪いため、既定はオンラインTCGの標準であるチェスクロック式を推す。

**部屋設定（ホストがロビーで選択・startで両者に配布）**:
- なし（既定・現状どおり）
- 公式風: 対戦全体30分。時間切れ＝両者敗北（公式準拠モード）
- 標準: 各プレイヤー持ち時間30分（チェスクロック）。切れたら切れ負け
- スピード: 各20分 ＋ 1手（自分が入力すべき場面）90秒。どちらか切れたら負け
- カスタム: 分数を指定

「厳しくするなら各ターンにも時間を」→ チェスクロック式なら**1ターン制限は必須ではない**
（長考は自分の残り時間を削るだけ）。ただし「1手上限」はテンポ維持に有効なのでスピード設定に同梱。

**実装方式（ロックステップ整合の要点）**:
- DO は入力保存時に ts を持っている → **broadcast にも ts を載せる**。
- 「時間を消費している側」は決定論的に分かる（手番側、防御プロンプト中はその決定側）。
  両クライアントが同じ ts 列から同じ残り時間を計算＝表示が一致し、Gを汚さない（hash対象外）。
- **時間切れの執行**: 切れたかどうかも ts から決定論的 → どちらのクライアントも
  `{t:'timeout', seat}` を送ってよい。適用時に各クライアントが ts から再検証して
  `lose(seat,'時間切れ')`（不正クレーム不能・自爆待ち不要）。
- 切断中の扱い: peer切断イベント間はクロック停止（猶予）をオプション化。猶予超過は切断側の敗北。
- UI: TurnPill 付近に両者の残り時間。残り1分で警告色＋SE。

工数: 1〜1.5日（DO ts配布・クロック計算・設定UI・執行・テスト）。

## 3. 相手切断時の勝利宣言（★実は今いちばん困る穴）

相手がタブを閉じて戻らないと、残った側は「無限に待つ」か「自分が投了」しかない。
DO は接続状態を知っている → 相手切断が N 分（例3分）続いたら残存側に
「相手の接続が切れています — 勝利して退室する」ボタンを表示 → `forfeit` を切断側名義で
DO が代理発行（`{t:'input', seat:切断側, d:{t:'forfeit'}}` を DO 自身が採番・配信）。
持ち時間機能を入れるならその切断猶予と一本化できる。工数: 半日。

## 5. 手番通知・スリープ防止

- 自分の入力待ちになったら: 通知音（既存SE流用）＋ `document.title` 点滅＋（許可時）バイブ。
- 対戦中は Screen Wake Lock API で画面スリープ防止（iOS Safari 16.4+対応）。
工数: 半日。

## 6. 戦績記録（D1）

`matches(id, host_uid, guest_uid, winner_seat, reason, host_leader, guest_leader, turns, game_no, ended_at)`。
realtime Worker に D1 バインドを追加し、両クライアントから `{t:'result', winner}` が一致したら書く
（またはDOが forfeit/timeout は自前で確定）。ロビー/ホームに「オンライン戦績 12勝8敗（リーダー別）」。
工数: 1日。

## 7. リプレイ再生

終局時に {seed, decks, 入力ログ} を D1/R2 へ保存（今はDOのTTLで消える）。
「リプレイを見る」＝ welcome 復帰と同じ経路を実速（sleep有効）で再生し、任意で倍速。
ロックステップの構造上ほぼタダで手に入る差別化機能。工数: 1〜1.5日（保存先とUI）。

## 8. エモート/定型チャット

定型文のみ（よろしく/ナイス/考え中/ごめん、もう一回! 等8種）＋スタンプ。
`{t:'emote', id}` を input と別チャネルで中継（ゲーム状態外・hash無関係）。レート制限あり。工数: 半日。

## 9. desync 自動復旧

現在: 検知→対戦中断。改善: 検知したら両クライアントが自動で「welcome 完全リプレイ」を実行して
同一状態を再構築（サーバの入力ログが正）。リプレイ経路は実装済みなので、発火条件と UI だけ。
懸念: 原因バグを覆い隠すので、dump 送信（現行のデバッグ計器）は残したまま「復旧したが記録は残る」に。
工数: 1日。

## 10. 部屋オプション拡充

- 先攻の決め方: ランダム（現状）/ ホスト指定 / 勝者後攻（リマッチ時）
- リマッチ時に先攻を入れ替える（既定ON推奨）
- 実装: room 設定オブジェクトを ready 前に `{t:'config', ...}` でホストから送り、start に同梱。
  `G.firstPref` は 'me'/'cpu' 指定が既にエンジンにあるため、start の firstPref を両者同値で流すだけ。

## 実装順の推奨

**第1弾（事故予防と要望・計2日強）**: ①版数チェック → ②持ち時間（切断猶予込み）→ ③切断時勝利宣言（②と統合）→ ④招待リンク
**第2弾（楽しさ・計2日強）**: ⑤手番通知/Wake Lock → ⑥戦績 → ⑧エモート
**第3弾**: ⑦リプレイ → ⑨desync自動復旧 → ⑩部屋オプション
