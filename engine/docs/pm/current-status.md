- プロジェクトの3法則（台帳から再確認）:

## 4. 次の実験候補（たたき台・優先度順）

どれを走らせるにも「何が分かったら成功か」を先に固定する。以下はその素案。

### 候補1: Hybrid live の実測（費用: APIキーのみ／半日）— 私の推し
- 仮説: per-matchup の live Claude 戦略シェイピングは puct を上回る（静的プロファイルは✗だが、非対称マッチで効く兆候あり=E25）。
- 成功条件: `llm-warm-cache`→`OPCG_LLM_CACHE` 決定的再生の measure-matchup（N≥40）で hybrid 対 puct が符号検定 p<0.05。負けたら「LLM戦略層は不要」で確定クローズできる。
- 根拠: 手順・配管は全部出荷済み（`docs/ai-design.md` §10.4）。**最小費用で「劇的に強く」の残レバーを1本消化でき、negativeでも記事ネタになる**。

### 候補2: クラウド段階B（費用: 数万円／数日・要予算合意）
- 仮説: value↔探索の鶏と卵は「多世代×大規模」で解ける＝世代を追うと「学習vs手作り」ptが −→0→+ へ動く。
- 成功条件/撤退基準: `docs/cloud-setup.md` に明記済み（兆しが無ければ段階Cへ進まない）。
- 注意: 改善する保証なしと文書化済み。**予算上限と撤退判定の世代数を事前にみちるさんと合意してから**。

### 候補3: lucy/ace/nami 向けの資源項目探索（費用: 0／1セッション）
- 仮説: ドン差が teach/hancock に効いたように（E27）、残る3リーダーにも固有の資源項目（手札の質・ライフ管理等）が evalState に存在する。
- 成功条件: measure-matchup ミラーで対象リーダー改善（目安 p<0.1）・他リーダー退行なし（リーダー別適用が前提）。
- 仮説源: みちるさんのプレイ観察を先に集める（E33方式）。観察なしの総当たりはやらない。

### 候補0（実験でなくガードレール・次セッション冒頭に実施推奨）: 実験出力と出荷物の分離
- E32の再発防止: az-loop.sh 等の実験系ツールが `src/` を直接書かないよう出力先を分離し、採用時のみ明示コピーする。安価で確実に事故を1類型消せる。
