# カード効果 完全実装ワークフロー（audit駆動）

全カードの効果を公式カードリスト準拠で漏れなく実装するための作業手順書。
実装セッション（Opus等）はこの文書と `engine/CLAUDE.md` を読んでから作業すること。応答は日本語・簡潔。

---

## 0. 役割とゴール

あなたは、ワンピースカードゲーム対戦シミュレーターに、カード効果を**弾（セット）単位で、公式サイトの情報に基づいて漏れなく実装する**エンジニアです。
現状の主なギャップは `tools/audit-report.json` に機械的に列挙されている（2026-07 初回集計: トリガー未実装525 / テキスト不一致84 / フィールド不一致6 / DB未収録2 / トリガー過剰1）。これを**ゼロにする**のがゴール。

## 1. 正本と道具（照合はスクリプトが済ませてある）

**重要**: 公式HTMLではトリガーが `<div class="trigger">` という**本文と別のブロック**にあり、旧スクレイパーが取り逃していた。`cards.js` の `text` にトリガー句が無いのはこのため。**トリガーの正本は official-full.json の `trigger` フィールドのみ**。

## 2. 絶対原則（例外なし）

1. **効果の正本は official-full.json**（公式サイトの機械取得スナップショット）。カードを実装する前に必ず該当エントリを読む。近似・推測・記憶・デッキ解説記事の孫引きからの実装は**禁止**。過去に近似実装が実バグを生んだ（ラフィット、おれの時代だ!!!!、ドン付与タイプの誤り）。
2. **公式Q&A照合を必須化（裁定の正本その2）**。text確認の**直後**に `node tools/qa-lookup.js <番号>` で該当カードの公式Q&Aを読む（キャッシュ `tools/official-qa.json`。無い新弾は `node tools/scrape-qa.js` を再実行）。裁定が挙動に効くQ&Aはテストケース化する。実対戦で発覚したバグの多くは裁定未確認が原因。
3. **スナップショットが疑わしい場合のみ** onepiece-cardgame.com/cardlist/ を直接確認し、必要なら `node tools/scrape-official-full.js` で再生成する。公式が取得できない場合のみ cardrush-op.jp 等で補う。
4. **日本語ソースのみ**。英語TCGサイト（limitlesstcg.com等）は使わない（海外は発売時期・カードプール・裁定が異なる）。検索は日本語で。
5. **確定と推定を区別して報告する**。解釈が推定になったカードは `special`/コメントでフラグを立て、勝手に確定実装しない。
6. **既存のカード・テストを壊さない**。実装済みカードの挙動を変えない。編集後は必ずテストを回す（§7）。
7. `text` フィールドは公式テキストを**一字一句そのまま**使う。省略・言い換え禁止（現状84件の text_mismatch は手書き要約が原因。公式全文に置換していく）。

## 3. 作業単位とフロー（1セッション = 1〜2セット）

```
① node tools/audit-cards.js <セット>   … そのセットの未解決差分を確認
② 各カード: official-full.json の該当エントリを読む（text と trigger の両方）
③ node tools/qa-lookup.js <番号> で公式Q&Aを読む（正本その2。キャッシュに無い新弾は scrape-qa.js を再実行）
④ 効果を4要素に分解（§4）→ パターン網羅チェックリスト（§6）を通す → 既存opにマッピング → cards-fx.js に実装
⑤ tests/fx-cards.js（または unit-example.js）に回帰ケースを追加（フルフロー1本必須・裁定が効くQ&Aはテスト化）
⑥ cd engine && node tests/test.js が緑になるまで直す ＋ node tools/lint-fx.js <セット> で ERROR 0
⑦ node tools/audit-cards.js <セット> を再実行 → 差分ゼロを確認
⑧ ルートで node scripts/sync-engine.mjs → npm test / npm run build（web反映）
⑨ 報告（§8のフォーマット）
```

- 差分の優先順: **field_mismatch / missing_in_db（データバグ・即修正）→ trigger_unimplemented（本丸）→ text_mismatch（公式全文への置換）**。
- missing_in_db は `node tools/scrape-cards.js` と `node tools/scrape-attributes.js` の再実行で解消（cards.js を再生成したら def 上書きとの整合を audit で再確認）。

## 4. 効果の4要素分解（すべての効果をこの形で実装）

1. **タイミング**（いつ）: 登場時/アタック時/ブロック時/KO時/ターン1回/永続/起動メイン/トリガー/ターン終了時 など → fx のキー（`onPlay`/`onAttack`/`onBlock`/`onKO`/`trigger`/`static`/`act` 等。全フック一覧は `docs/opcg-effect-system-design.md` §12）
2. **条件**: 「リーダーが〇〇の場合」「ドン‼がN枚以下の場合」「ライフN枚以下の場合」など → `cond`
3. **コスト**: ドン‼-N / 手札を捨てる / ドンをレスト など → コストop（`{op,..,then}`型）。**任意（〜できる/〜まで）か強制かを必ず区別**（任意=`optional:true`、「〜まで」=0個可）
4. **効果**: ドロー/KO/パワー増減/バウンス/ドン付与 など → 対象・数・**期限**（このターン中/このバトル中/次の相手エンドフェイズ終了時まで…）を正確に

## 5. トリガー実装の方針（525件＝最大の作業）

audit の分布ではトリガーは高度にテンプレート化されている（上位: 「このカードの【メイン】効果を発動する」59 /「このカードを登場させる」51 /「カードN枚を引く」34 /「コストN以下レスト」22 /「コストN以下KO」22 /「N枚引きN枚捨て」20 …）。

- **テンプレート優先**: 既存の `fx.trigger` 実装例（51件・cards-fx.js を grep）に倣い、頻出型から一括で埋める。
  - 「このカードの【メイン】効果を発動する」→ イベント本体の `main.fx` を trigger から再利用（既存パターンを踏襲。重複記述しない）
  - 「このカードを登場させる」→ 既存の登場系 trigger op（該当既存カードの書き方に合わせる）
- トリガーは**任意発動**（人間には確認プロンプト・`askTrigger` 経由）。コスト0で発動。【バニッシュ】被弾時は発動不可（エンジン側処理を確認）。
- 実装したら `tests/fx-cards.js` にセット単位の trigger 回帰を追加。

## 6. 頻出する実装ミス（必ず回避。過去に実際に発生）

`engine/CLAUDE.md` §4/§8/§9 が正本。特に:

- **ドン‼の状態（最重要）**: テキストが「レストのドン‼を付与」なら必ず `donAttach`/`donAttachAll`（`P.don.rested` から）。アクティブから付与するのは手動付与だけ。「ドン‼-N」は必ず `donMinus`（ドンデッキへ戻る・donTotal減）、「ドンN枚をレスト(①②…)」だけ `{don:N}`(payDon)。
- **「パワーN以下」（現在パワー=`maxEffPower`）と「元々のパワーN以下」（`maxPower`）の取り違え**。コストも同様（`maxCost`=効果込み / `maxBaseCost`=元々）。
- **フィルタはopのトップレベルに置かない**（`op.filter` 内に置く。トップレベルは無視される）。
- **「〜できる」「〜まで」= `optional:true` の付け漏れ**（系統的多発）。
- **カード名と特徴の取り違え**（「」=カード名、《》=特徴。「を含む特徴」は `traitIncludes`、完全一致は `trait`）。
- **「このキャラのコスト±N」は `staticCost`**（盤面・除去耐性）。手札のプレイコスト軽減だけ `costMod`。
- **追加コストの見落とし**（例: 神避 OP13-076 はドン5レストが必要。0コストではない）。**手札コスト**（例: OP15-060 エネルのブロッカー獲得は手札1枚捨て）。
- **効果の期限**（このターン中/このバトル中/次の相手エンドフェイズ終了時まで/次の自分のターン開始時まで）を永続と混同しない。
- **リーダー効果を余計なゲートで止めない**（実際の発動条件を満たす限り毎ターン確実に発動。存在しない条件を勝手に足さない）。
- **コスト・パワー等の数値を名前の印象で思い込まない**（例: 「神官」ゲダツ OP15-063 はコスト6）。official-full.json の値をそのまま使う。
- **実装前レビューは「パターン網羅チェックリスト」を必ず通す**（正本 `.claude/skills/opcg-card-implement/SKILL.md`・要点は `engine/CLAUDE.md` §11）。骨子: 対象候補0/1/複数（候補1件の自動選択）／任意vs強制／リーダーを対象に含むか／条件不成立時はUI抑止・once未消費／ライフ端点0/1/2（裁定集中）／バトルフロー上の反映位置／コスト支払い不能=発動不可／isCPU分岐とside／テストは必ずフルフロー／実装後のQ&A照合→テスト化。

## 7. 人間プレイヤーとCPUの発動仕様

- 【登場時】【起動メイン】は**人間には自動発動させない**（手動クリックで発動）。CPUは自動解決。
- 任意効果（「〜できる」「〜まで」）は選択可能に（発動する/しない・対象選択）。強制効果は自動解決。空対象でフリーズしないこと（候補0→即null解決）。
- 【ターン1回】は回数管理（既存の `once` ゲート/共有フラグに倣う）。
- 複数同時誘発はバトルフックの順序（`declareAttack` 固定順）に従う。トリガーの発動可否は人間に確認する。

## 8. わからない効果への対処と報告フォーマット

- 解釈不明 → 公式の総合ルール・公式Q&A(FAQ)を調査（まず `node tools/qa-lookup.js --search <kw>` / `--rules`）→ 既存の効果パターンに照らして最も整合的な実装を選ぶ → それでも不明なら**推定フラグを立てて報告**（勝手に確定実装しない）。
- セッション終了時の報告:
  - **確定実装カード**（No./名前/効果要約）
  - **推定・要確認カード**（理由付き）
  - **未実装カード**（理由付き）
  - audit 再実行後の残差分数（セット別）

## 9. テスト合格条件（engine/CLAUDE.md §5）

- `cd engine && node tests/test.js` 緑（CPU対CPU: noWinner=0 doubleAttacks=0 / 人間オートパイロット: 真の停止0・クリック不能プロンプトなし を含む9ステップ）
- `node tools/lint-fx.js <セット>` で **ERROR 0**（存在しないop名/フック名ゼロ。WARNは1件ずつトリアージ）
